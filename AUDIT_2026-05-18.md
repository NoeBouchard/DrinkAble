### ⚠️ Real bug found while pinging the live advisor
With a contrived "all candidate shops are closed Monday afternoon" payload, the Anthropic model didn't follow the JSON schema — it stuffed its apology *into the `shopName` field*. The server falls back to slugifying the unmatched name as `shopId` ([src/server/advisor-handler.js:180](src/server/advisor-handler.js:180)), producing a junk recommendation card:

```
shopId: "i-apologize,-but-all-three-coffee-shops-in-the-provided-list-..."
shopName: "I apologize, but all three coffee shops in the provided list ..."
```

The UI would render that card, and its "Show on map" button would route to a non-existent shop. Low probability on a real 45-shop dataset (something is usually open) but **not zero** — and if a tester sees it once, it looks broken. Fix is server-side: drop recommendations whose `shopName` doesn't match a candidate, and fall back to "advice only" if nothing remains.

## 6. First-tester readiness — honest take

**Would a London friend understand it in <10s?** **Yes.** Splash is clean: name + one-line value prop + Get-started. The value prop ("Personalized, conversational coffee recommendation from a knowledgeable expert") is wordy and slightly buzzword-y — "Find your next coffee" (already the meta-description) is sharper — but no friend will miss the point.

**Would they get a useful recommendation in <60s?** **Probably yes, just.** Splash → Q1 (~5s, must pick ≥1) → Q2 (~5s, skippable) → Location (1–2s if they grant, instant if they decline) → FirstRecommendation Claude call (4–8s on a warm function, longer on a cold start). Total ≈ 30–45s on desktop. On mobile 4G with a cold-start serverless function, it could push past 60s — the Claude latency is the long pole.

**Single biggest friction point:** **The Q2 priorities don't actually do anything structured.** This is the credibility crack:

- Onboarding asks "Roaster-owned" / "Recognized globally (top 100)" / "Certified specialty-grade" with the implicit promise that the recommendation will respect them.
- The data layer doesn't carry those signals (§3 — 0/45 shops). The Q2 selections are passed as **free text** to the LLM and it has to guess from prose.
- For a friend who picks "Roaster-owned" + "Recognized globally (top 100)" and then sees an explanation that doesn't *name* those dimensions back to them, the onboarding will feel decorative. *Why did you ask if you don't use it?*

Two secondary risks:
1. **4–8s Claude latency on FirstRecommendation** with a spinner. Acceptable on first run because the screen sets expectations ("Asking the Advisor for the perfect spot near you…"), but feels like an eternity on repeat queries from AdvisorHome.
2. **Malformed-recommendation bug** I found while testing (§2 — LLM stuffs apology into `shopName`, server slugifies it into a garbage `shopId`). Low probability on real data but a one-shot credibility kill if it fires.

### What I'd build next, in priority order

1. **Add the three structured fields to all 45 shops** (`roasts_own_beans: bool`, `world_top_100: bool`, `specialty_certified: bool`). One-shot data pass — Claude can do it from the existing `specialtyFocus`/`description`/`roaster` strings + a quick web verification per shop. Then have the advisor-handler **prefilter** candidate shops by Q2 priorities before sending the top-10 to the LLM. Closes the onboarding-promise gap, and tightens the LLM context so its picks get noticeably more relevant.
2. **Harden advisor output validation** server-side: drop recs whose `shopName` doesn't match a candidate, fall back to advice-only if nothing remains.
3. **Document `ANALYTICS_WEBHOOK_URL`** in `.env.example` and confirm the Apps-Script sheet is actually wired in Vercel prod env.
4. **Confirm Vercel auto-deploy is on**, then either update DEPLOYMENT.md to match or remove the manual section.
5. *(Optional, deferred)* Session 6 DB expansion to 200–300 shops. Don't do this before #1 — adding 5× more shops to an LLM-only personalization layer multiplies the credibility risk.

If you ship #1 + #2 in a tight Session 6, you have a defensible product to put in front of a London friend.
